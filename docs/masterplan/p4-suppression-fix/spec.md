# v4.1.1 — P4 Suppression Fix Design Spec

## Goal

Address the two findings from the codex adversarial review of v4.1.0 (commit `bbe5a38`):

1. **HIGH** — v4.1.0's TaskCreate projection claims to silence the reminder, but smoke evidence (events.jsonl line 37 of v4-lifecycle-redesign) shows the reminder fires with tasks present. Suppression is keyed on **recent Task\* tool usage**, not task existence.
2. **MEDIUM** — Step C drift recovery is documented as every-entry reconciliation but only runs on the first entry per session (commands/masterplan.md:1376).

Ship v4.1.1 with a verified per-state-write `TaskUpdate` priming strategy that actually silences the reminder during Step C execution, plus a create-once / reconcile-every-entry split in Step C.

## In-session empirical baseline

The reminder fired once during this very brainstorm session (events.jsonl `empirical_observation_in_session`) after ~4 tool calls with zero `Task*` usage. This is a concrete, in-the-wild baseline observation that aligns with the codex finding: with no recent `Task*` activity, the reminder fires regardless of whether tasks exist.

## Relationship to v4.1.0 mechanism (additive, not replacement)

v4.1.0 already mirrors `state.yml` task-transitions to `TaskUpdate` (commands/masterplan.md:1378). Transitions covered: `current_task` advance, wave dispatch begin, wave-member digest, `pending_retro` flip, `complete` flip, `blocked` write.

What v4.1.0 does NOT cover — and what v4.1.1 adds:
- **Idle-turn gaps between transitions.** When the orchestrator does multiple sequential tool calls (Read, Bash, Edit, AskUserQuestion) within a single task without flipping `current_task` or wave state, no `TaskUpdate` fires. The per-turn reminder-suppression decay outruns the transition cadence. This is the exact failure mode codex flagged and we observed in-session.
- **First entry into Step C before any transition has happened yet.** The rehydration call covers session-start; nothing between rehydration and the first transition.

v4.1.1 closes both gaps by promoting the touch from "per-transition" to "per state.yml write" — the canonical material-action point per CD-7. Every state write is by definition a moment the orchestrator considers something worth persisting, which makes it the right priming pulse cadence.

## Design decisions

### 1. Suppression mechanism: per-state-write `TaskUpdate` touch

| Axis | Decision |
| --- | --- |
| Touch point | **Every `state.yml` write** by the orchestrator |
| Touch shape | `TaskUpdate(current_task, status=in_progress)` — idempotent re-stamp |
| Scope | **Step C execution only** (brainstorm/plan/halt-gate reminder firings are appropriate) |
| No-current_task | **Skip silently** when `state.current_task == ""` |

Why "every state write": canonical material-action points (CD-7 single-writer rule) — wave dispatch, Step C entry, task transitions, gate close-outs. Skips reply-only turns. Bounded tool cost (~5–15 touches per run).

### 2. Step C drift recovery split

**New state field:** `step_c_session_init_sha` (string, default `""`). Stored in `state.yml` alongside `phase` / `current_task`.

**Session signature:** empirically `${CLAUDE_SESSION_ID}` is **NOT set** in the current Claude Code runtime (confirmed in this brainstorm session — `printenv CLAUDE_SESSION_ID` returned nothing). The signature must therefore be a UUID generated by `bin/masterplan-state.sh` on first Step C entry of the session and persisted into `state.yml`. The helper writes the UUID into `step_c_session_init_sha` atomically alongside the existing state-write path. The runtime envvar path is retained in spec as a forward-compat fallback only — if a future Claude Code build sets `CLAUDE_SESSION_ID`, the helper prefers it; otherwise it generates a v4 UUID. Either way, the orchestrator never tries to read the env var directly; the helper is the single source of truth.

**On Step C entry:**

```
if state.step_c_session_init_sha == "" OR state.step_c_session_init_sha != <current-session-signature>:
    # First entry of this session
    full TaskCreate rehydration (current v4.1.0 behavior)
    state.step_c_session_init_sha = <current-session-signature>
    TaskUpdate touch
    record `step_c_init_complete` event
else:
    # Subsequent entry in same session
    drift_check()
    TaskUpdate touch
    record `step_c_drift_check_complete` event
```

**Drift check scope:** current_task + status counts.
- Verify: TaskList task tagged in-progress matches `state.current_task`
- Verify: `in_progress count == 1` if state.phase is mid-wave; `pending count > 0` if waves remain
- Correct via `TaskUpdate` calls for any drift
- Append `step_c_drift_detected` / `step_c_drift_corrected` events

## Open items / risks

- **R1: Decay rate (pre-registered rescope path).** Per-state-write touch frequency may still be too sparse if reminder-suppression decays faster than the inter-write turn count. Stage 2 smoke measures this directly. **If smoke fails (reminder fires on any state-write turn), the fallback is pre-registered, not improvised:** redesign to Option D — augment per-state-write touches with an explicit idle-turn heartbeat (a `TaskUpdate` of the current task at the top of every Step C turn loop iteration that has NOT yet performed a state write). This raises the touch rate to ≥1/turn under any circumstance. If even that fails, the honest rescope is to drop the suppression claim entirely and ship v4.1.1 as "drift-recovery + amended docs" only.
- **R2: TaskUpdate on missing task.** `TaskUpdate(current_task, …)` requires `current_task` to map to a TaskList entry. Edge case: resume-from-compaction where state.yml has `current_task` set but the task was never created in the new session. Mitigation: detect "TaskUpdate failed: task not found" and trigger drift-recovery rehydration as the recovery path. (Already implied by the create-once-per-session gate — `step_c_session_init_sha == ""` for a fresh session means full rehydration runs first.)
- **R3: Opaque suppression rule.** Docs investigation may turn up that the suppression heuristic is internal/undocumented. In that case the smoke test becomes the only ground truth; if it fails, the honest path is to rescope and drop the suppression claim. Don't ship v4.1.1 with the same unverified claim v4.1.0 had.
- **R4: Session ID stability.** If `${CLAUDE_SESSION_ID}` isn't stable across compaction (compaction may produce a fresh session ID), `step_c_session_init_sha` would not match and full rehydration would run on every entry, defeating the optimization. Mitigation: also store a fallback UUID generated by `bin/masterplan-state.sh` on first init; compare against either. Decide concretely during the plan phase based on empirical behavior.
